# Deployment Guide — SalesFlow CRM

Deploy the app to an Ubuntu server. Each instance gets its own database, systemd service, and nginx config — multiple instances can coexist on the same server.

## Stack

| Component     | Technology                       |
| ------------- | -------------------------------- |
| OS            | Ubuntu 22.04 / 24.04             |
| Database      | PostgreSQL 16 + pgvector (local) |
| Runtime       | Node.js 20 LTS + pnpm            |
| Process       | systemd                          |
| Reverse proxy | Nginx                            |
| SSL           | Let's Encrypt (certbot)          |

## Prerequisites

1. **An Ubuntu server** — Minimum recommended: 2 GB RAM / 1 vCPU. For heavier AI/LLM workloads or many concurrent users, use 4 GB+.

2. **DNS configured** — Create an A record pointing your domain to the server's IP address. Allow time for DNS propagation before running the script.

3. **SSH access** — `ssh root@<SERVER_IP>`

4. **LLM API credentials** ready (optional — only needed for AI features).

## Quick Start

```bash
# 1. Copy script to server
scp deploy.sh root@<SERVER_IP>:/root/deploy.sh

# 2. SSH in
ssh root@<SERVER_IP>

# 3. Run with your domain
chmod +x /root/deploy.sh
./deploy.sh crm.example.com
```

The script will interactively prompt for:
- Certbot email (required for SSL)
- GitHub repo URL (defaults to Sendrato/sendrato-salesflow)
- Git branch (defaults to main)
- LLM/Forge API credentials (optional, for AI features)

Everything else is auto-derived from the domain name:

| Resource          | Derived from `crm.example.com`         |
| ----------------- | -------------------------------------- |
| App directory     | `/opt/crm.example.com`                 |
| System user       | `crm-example-com`                      |
| PostgreSQL DB     | `crm_example_com`                      |
| PostgreSQL user   | `crm_example_com`                      |
| Systemd service   | `crm-example-com.service`              |
| Nginx config      | `/etc/nginx/sites-available/crm-example-com` |
| Port              | Auto-assigned (next free from 3000)    |

The script is idempotent — re-running it for the same domain will pull the latest code and redeploy.

## Multiple Instances

To deploy a second instance on the same server, simply run the script again with a different domain:

```bash
./deploy.sh crm.acme.com
./deploy.sh crm.other.com
```

Each instance gets its own port (auto-assigned), database, service, and nginx config. They share the same PostgreSQL server and Node.js installation.

## First Login

After deployment, visit `https://<your-domain>`. Since no users exist yet, the app will show a **"Create admin account"** form. The first user to register automatically becomes the admin.

Additional users can only be created by the admin via **Settings → Invite User**.

## Environment Variables

Auto-generated by the script and written to `/opt/<domain>/.env`:

| Variable                      | Description                                               |
| ----------------------------- | --------------------------------------------------------- |
| `DATABASE_URL`                | Auto-generated (instance-specific database)               |
| `JWT_SECRET`                  | Auto-generated (random 32-byte secret)                    |
| `PORT`                        | Auto-assigned (next free port from 3000)                  |
| `BUILT_IN_FORGE_API_URL`      | LLM/Forge API base URL (prompted during setup)            |
| `BUILT_IN_FORGE_API_KEY`      | LLM/Forge API key (prompted during setup)                 |
| `VITE_FRONTEND_FORGE_API_KEY` | Frontend Forge API key (prompted during setup)             |
| `VITE_FRONTEND_FORGE_API_URL` | Frontend Forge API base URL (prompted during setup)        |

**Note:** `VITE_*` variables are baked into the frontend at build time. If you change them after the initial build, you must rebuild:

```bash
cd /opt/<domain>
pnpm run build
systemctl restart <service-name>
```

## What the Script Does

1. **System update** — Installs base packages (curl, git, build-essential, ufw)
2. **Firewall** — Enables UFW, allows SSH and Nginx (80/443)
3. **PostgreSQL + pgvector** — Installs (if needed), creates instance-specific database + user with random password, enables `vector` extension
4. **Node.js 20 + pnpm** — Installs via NodeSource (if needed)
5. **Application** — Creates system user, clones repo to `/opt/<domain>`
6. **Environment** — Writes `.env` with auto-generated credentials and prompted LLM settings
7. **Build** — `pnpm install`, `pnpm run build`, `pnpm run db:push` (migrations)
8. **systemd** — Creates instance-specific service that auto-starts and restarts on failure
9. **Nginx + SSL** — Reverse proxy to the assigned port with streaming support, obtains Let's Encrypt certificate

## Operations

Replace `<service>` with your service name (e.g. `crm-example-com`).

### Service management

```bash
systemctl start <service>
systemctl stop <service>
systemctl restart <service>
systemctl status <service>
```

### View logs

```bash
# Follow live logs
journalctl -u <service> -f

# Last 100 lines
journalctl -u <service> -n 100

# Since last hour
journalctl -u <service> --since "1 hour ago"
```

### Deploy updates

```bash
cd /opt/<domain>
git pull origin main
pnpm install --frozen-lockfile
pnpm run build
pnpm run db:push          # apply any new migrations
systemctl restart <service>
```

### Database access

```bash
# Credentials are in /opt/<domain>/.env
sudo -u <app-user> psql <database>
```

### SSL certificate

Auto-renews via certbot's systemd timer. Verify:

```bash
systemctl list-timers | grep certbot
certbot renew --dry-run
```

### Backup database

```bash
sudo -u postgres pg_dump <database> > /root/<database>-backup-$(date +%Y%m%d).sql
```

## Troubleshooting

| Issue                        | Check                                                                                              |
| ---------------------------- | -------------------------------------------------------------------------------------------------- |
| App won't start              | `journalctl -u <service> -n 50` — look for missing env vars or DB connection errors                |
| 502 Bad Gateway              | Is the app running? `systemctl status <service>`. Is it on the right port? Check `PORT` in `.env`  |
| SSL cert failed              | Is DNS pointing to this server? `dig <domain>`. Is port 80 open? `ufw status`                      |
| DB migration fails           | Check `DATABASE_URL` in `.env`. Test with `psql <connection-string>`                                |
| Build fails                  | Check Node/pnpm versions: `node -v`, `pnpm -v`. Ensure enough RAM (2 GB minimum)                   |
| VITE\_\* changes not visible | These are compile-time. Rebuild: `pnpm run build && systemctl restart <service>`                    |
| Port conflict                | Check assigned port in `.env`. List all ports: `grep -r '^PORT=' /opt/*/.env`                       |
